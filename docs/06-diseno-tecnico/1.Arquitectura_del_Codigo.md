## 2. Estructura del Repositorio Backend (Spring Boot)

Dado que la Fase 5 decidió usar un **Modular Monolith**, la estructura del proyecto Java no estará agrupada por "capas tontas" (e.g. todos los controllers juntos), sino que usará **Domain-Driven Packaging** (*Package by Feature/Component*). Cada módulo de la Fase 4 es un paquete raíz.

```text
backend-monolith/
├── src/main/java/com/finca/
│   ├── FincaApplication.java        # Punto de entrada (main)
│   ├── core/                        # Utilidades globales y Configuración base
│   │   ├── config/                  # Cors, Swagger, RestTemplate beans
│   │   ├── exceptions/              # GlobalExceptionHandler (D7)
│   │   └── security/                # JwtFilters, SecurityConfig (D4)
│   │
│   ├── modules/                     # LOS CONTEXTOS AISLADOS (Phase 4 Módulos)
│   │   ├── auth/                    # MOD-AUTH
│   │   │   ├── domain/              # Entidades, Enums
│   │   │   ├── api/                 # Controllers (REST)
│   │   │   ├── application/         # Use Cases / Services
│   │   │   └── infrastructure/      # JpaRepositories
│   │   │
│   │   ├── booking/                 # MOD-RSV (Core Transaccional)
│   │   │   ├── domain/              # Reserva, SoftLockExpirationEvent
│   │   │   ├── api/                 
│   │   │   └── infrastructure/      # Repositorios con SELECT FOR UPDATE
│   │   │
│   │   ├── payment/                 # MOD-PAY
│   │   ├── catalog/                 # MOD-SRCH, MOD-FCNT, MOD-REV
│   │   └── notification/            # MOD-NOT (Event Listeners)
│   │
│   └── shared/                      # Interfaces públicas permitidas entre módulos
│       └── events/                  # Eventos de dominio (ej. BookingCreatedEvent)
│
├── src/main/resources/
│   ├── application.yml              # Configuración de base de datos y Cloud
│   └── db/migration/                # Scripts de Flyway/Liquibase
└── pom.xml / build.gradle           # Dependencias
```

## 3. Estructura del Repositorio Frontend (Next.js)

Para maximizar la escalabilidad, usaremos el **App Router** de Next.js combinado con la filosofía **Feature-Sliced Design (FSD)**. Esto evita la carpeta `components/` gigante y agrupa la lógica de UI por módulo funcional.

```text
frontend-app/
├── src/
│   ├── app/                         # Next.js App Router (Páginas y Routing)
│   │   ├── (auth)/                  # Route Group para login/register
│   │   ├── (propietario)/           # Dashboard privado B2B
│   │   ├── fincas/                  # [id]/page.tsx (Ficha de la finca)
│   │   ├── layout.tsx               # Root Layout (Navbar global)
│   │   └── page.tsx                 # Home Page (Buscador)
│   │
│   ├── features/                    # Feature-Sliced Design (Módulos)
│   │   ├── auth/                    # Lógica de MOD-AUTH
│   │   │   ├── components/          # LoginForm, RegisterForm
│   │   │   ├── hooks/               # useAuth
│   │   │   └── api/                 # Llamadas fetch/axios
│   │   │
│   │   ├── booking/                 # MOD-RSV y MOD-PAY
│   │   │   ├── components/          # CheckoutModal, DatePicker
│   │   │   ├── store/               # Zustand/Redux slice de la reserva
│   │   │   └── api/
│   │   │
│   │   └── catalog/                 # MOD-SRCH
│   │       ├── components/          # FincaCard, Filters
│   │       └── hooks/               # useSearchFincas
│   │
│   ├── shared/                      # Componentes estúpidos y globales (Design System)
│   │   ├── ui/                      # Button, Input, Modal, Spinner (Tailwind)
│   │   ├── lib/                     # utils.ts, cn() para Tailwind
│   │   └── api/                     # Instancia base de Axios/Fetch
│   │
│   └── styles/
│       └── globals.css              # Tailwind directives
│
├── public/                          # Assets estáticos
├── next.config.mjs
├── tailwind.config.ts
└── package.json
```
