# Entregable 10 (D10): Consolidación ADR

**Proyecto:** Nos Fuimos de Finca
**Fase:** 5 — Diseño Arquitectónico
**Estado:** Aprobado

### Registro de Decisiones de Arquitectura

Este documento indexa las decisiones técnicas irrevocables tomadas durante la Fase 5. Ningún desarrollador puede contravenir estas reglas durante las Fases 6 y 7 sin escalar un rediseño de la arquitectura.

| ID ADR | Área de Decisión | Enfoque Seleccionado | Justificación Clave |
| --- | --- | --- | --- |
| **ADR-001** | Deployment Topology | Híbrido: **Vercel (Front) + AWS (Back y DB)**. | Vercel maximiza el performance de SSR (Next.js), mientras AWS provee la infraestructura madura para base de datos relacional (RDS) y escalabilidad del backend (Beanstalk/ECS). |
| **ADR-002** | System Decomposition | **Modular Monolith** en Spring Boot. | Reduce latencia inter-módulo (0ms en llamadas a memoria) y simplifica drásticamente el manejo de transacciones ACID y Rollbacks frente a una arquitectura de Microservicios. |
| **ADR-003** | Architectural Style | Externo: **REST API** (JSON). Interno: **Event-Driven** (ApplicationEvents). | REST garantiza máxima compatibilidad con cualquier cliente B2C. Eventos internos desacoplan módulos evitando cuellos de botella por notificaciones y correos. |
| **ADR-004** | Communication Pattern | **Síncrono para lecturas/escrituras primarias. Asíncrono vía Webhooks y Tareas Background para 3eros.** | El backend debe aceptar de Wompi la orden, encolar el correo asíncronamente y devolver HTTP 200 rápido (Idempotencia). |
