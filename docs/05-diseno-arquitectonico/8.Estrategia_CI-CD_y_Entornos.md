# Entregable 8 (D8): Estrategia CI-CD y Entornos

**Proyecto:** Nos Fuimos de Finca
**Fase:** 5 — Diseño Arquitectónico
**Estado:** Aprobado

## 1. Topología de Entornos
El ecosistema completo (`Front + Back + DB`) existirá en tres ambientes aislados:
- **`dev`:** Ambiente para uso diario de los desarrolladores. Conectado a servicios de Sandbox (ej. Wompi Sandbox). Base de datos efímera.
- **`staging`:** Espejo del ambiente de producción. Usado para pruebas QA, UAT y validación final antes del lanzamiento.
- **`production`:** Entorno vivo con usuarios reales, conectado a pasarelas reales y con políticas estrictas de copias de seguridad de BD.

## 2. Estrategia de Control de Código (GitFlow Simplificado)
- **Ramas Principales:**
  - `main` (Despliega automáticamente a `production`).
  - `develop` (Despliega automáticamente a `staging`).
- **Feature Branches:** Todo desarrollo (ej. `feature/m-03-reservas`) nace de `develop` y se mezcla vía *Pull Request (PR)*. Solo se aprueba si las Pruebas Unitarias pasan.

## 3. Pipeline CI/CD (GitHub Actions y Vercel)
- **Frontend (Vercel):** *Zero Config.* Vercel escucha el repositorio de GitHub. Cualquier push a `main` hace deploy a producción. Cualquier push a un PR genera una URL de previsualización (Preview Environment).
- **Backend (GitHub Actions -> AWS):**
  - **CI (Continuous Integration):** Al empujar código, GitHub Actions corre el `./mvnw test`, compila el proyecto y reporta cobertura.
  - **CD (Continuous Deployment):** Si el *push* fue a la rama `main`, empaqueta el artefacto en un contenedor Docker, lo sube a AWS ECR y actualiza el cluster de ECS/Elastic Beanstalk sin *downtime* (Rolling Update).
