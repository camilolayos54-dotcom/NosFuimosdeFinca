# Entregable 4 (D4): Decisión de Descomposición del Sistema

**Proyecto:** Nos Fuimos de Finca
**Fase:** 5 — Diseño Arquitectónico
**Estado:** Aprobado

## Registro de Decisión Arquitectónica (ADR-002)

### Contexto
Existen 3 Contextos Limitados principales (Transaccional, Catálogo e Identidad). Con las arquitecturas modernas, es tentador crear un microservicio para cada contexto (e.g. un servidor Node para Auth, un servidor Java para Reservas, etc.). Sin embargo, la escala operativa del proyecto actualmente es de un equipo técnico pequeño iniciando operaciones ("Startup Phase").

### Decisión
Se adoptará una arquitectura de **Monolito Modular Estricto (Modular Monolith)** para el Backend.
Todos los contextos limitados (Auth, Fincas, Reservas) residirán dentro del **mismo** repositorio de código y se compilarán en un **único artefacto ejecutable** (.jar / Docker Image) administrado por Spring Boot.

Se prohíbe explícitamente el aislamiento físico (Microservicios), pero se exige aislamiento lógico: los paquetes internos (`com.finca.booking`, `com.finca.catalog`) no deben cruzar límites transaccionales ni inyectar repositorios de otros contextos, deben comunicarse mediante interfaces o eventos de dominio en memoria (ApplicationEvents).

### Consecuencias
**Positivas:**
- **Zero Network Overhead:** Las llamadas entre el motor de búsqueda y las reservas se hacen en memoria (0ms), cumpliendo fácilmente con los NFRs de rendimiento.
- **Transaccionalidad Simple:** Es posible manejar transacciones de BD nativas complejas y hacer `Rollback` de todo sin recurrir a patrones complejos como *Saga Pattern* o *Two-Phase Commits*.
- **Despliegue unificado:** CI/CD drásticamente más simple. Una sola base de datos (Un esquema, múltiples tablas).

**Negativas / Riesgos:**
- El código puede convertirse en un "Big Ball of Mud" si no se respetan las interfaces internas (Se mitigará con reglas estrictas de empaquetado como ArchUnit en Java o validadores de dependencias).
- Si el módulo de búsqueda sufre un pico de tráfico masivo (DDOS), todo el monolito puede caer, afectando al módulo transaccional. (Se mitigará escalando horizontalmente *todo* el monolito tras un Load Balancer de AWS).
