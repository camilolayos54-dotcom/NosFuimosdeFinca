# Entregable 5 (D5): Selección de Estilo Arquitectónico

**Proyecto:** Nos Fuimos de Finca
**Fase:** 5 — Diseño Arquitectónico
**Estado:** Aprobado

## Registro de Decisión Arquitectónica (ADR-003)

### Contexto
El Frontend (Next.js) y el App Móvil (Futura) necesitan comunicarse con el Backend Monolítico, y los propios módulos internos del monolito necesitan comunicarse entre sí tras completar acciones críticas (ej. enviar correos después de reservar).

### Decisión
1. **Externo (Client-to-Server):** El contrato hacia el exterior será un **API RESTful**.
   - GraphQL se descarta por añadir sobrecarga cognitiva no requerida para modelos relacionales planos.
   - gRPC se descarta porque el cliente principal es un navegador web (JSON es el estándar).
   - Se aplicará nivel 2 de madurez Richardson (Uso correcto de verbos HTTP GET, POST, PUT, DELETE y Códigos de Estado).
2. **Interno (Module-to-Module):** El estilo arquitectónico será **Event-Driven Architecture (Local)**.
   - Usaremos eventos de dominio publicables (ej. `Spring ApplicationEvents`). Un módulo "publica" que algo pasó (`ReservaHardLockedEvent`), y el módulo de Notificaciones actúa como "Observer" ciego escuchando ese evento.

### Consecuencias
- Integración predecible, estándar de facto en la industria. Herramientas como Postman, Swagger/OpenAPI y Fetch API en React funcionan nativamente sin librerías pesadas (GraphQL Apollo, etc.).
- El desacoplamiento interno vía Eventos de Dominio asegura que si el servidor de correo se cae (SendGrid HTTP timeout), la transacción del bloqueo en Base de datos no se haga Rollback por accidente.
