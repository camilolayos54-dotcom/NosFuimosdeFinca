### 1. SSD: Webhook de Wompi y Promoción a Hard-Lock

Este diagrama modela el flujo asíncrono donde Wompi confirma que el pago fue exitoso, disparando la inmutabilidad de la reserva.

```mermaid
sequenceDiagram
    autonumber
    participant Wompi (External)
    participant API (Spring Boot Webhook)
    participant DB (PostgreSQL)
    participant EventQueue (RabbitMQ)

    Wompi->>API: POST /api/v1/pagos/webhook/wompi
    Note over Wompi, API: Payload incluye Signature Hash
    
    API->>API: Validar Signature vs EventSecret
    alt Firma Inválida (Spoofing)
        API-->>Wompi: 401 Unauthorized
    else Firma Válida
        API->>DB: BEGIN TRAN
        API->>DB: UPDATE pago SET estado='aprobado' WHERE ref=?
        API->>DB: UPDATE reserva SET estado='hard_lock', expires_at=NULL WHERE id=?
        API->>DB: COMMIT
        
        API->>EventQueue: Publish Event (RESERVA_HARD_LOCK, reservaId)
        API-->>Wompi: 200 OK (Idempotente)
    end
```

### 2. SSD: Polling de Estado por parte del Frontend
Dado que el Webhook es server-to-server, el frontend móvil del turista debe enterarse del resultado.

```mermaid
sequenceDiagram
    participant Web (Next.js UI)
    participant API (Spring Boot)

    loop Cada 3 segundos en pantalla de carga
        Web->>API: GET /api/v1/pagos/referencia/{ref}
        API-->>Web: { estado: "pendiente" }
        Web->>API: GET /api/v1/pagos/referencia/{ref}
        API-->>Web: { estado: "aprobado" }
    end
    Web->>Web: Redirige a /pago/exito
```
