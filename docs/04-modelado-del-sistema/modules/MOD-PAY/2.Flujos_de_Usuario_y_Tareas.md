# Entregable 2 (D2): Flujos de Usuario y Tareas (MOD-PAY)

**Proyecto:** Nos Fuimos de Finca
**Fase:** 4 — Modelado del Sistema
**Módulo:** MOD-PAY (Pagos y Facturación)
**Estado:** Aprobado

### 1. Flujo de Tarea: Procesar Pago en Pasarela Externa (Turista)
**Source:** US-PAY-001, US-PAY-002

1. **[Inicio]** Turista viene redirigido desde el módulo M-03 (Reserva) con un `Soft-Lock` activo.
2. El sistema renderiza el Widget de Wompi (Drop-in) o redirige a la URL de Checkout de Wompi.
3. Turista ingresa datos de tarjeta de crédito o selecciona PSE.
4. Hace tap en "Pagar $200".
5. Wompi procesa la transacción internamente.
6. **[Decisión]** ¿Resultado de la transacción?
   - APROBADO: Redirige al Turista a `/pago/exito`. Muestra mensaje de celebración y detalles de reserva (Hard-Lock).
   - DECLINADO/ERROR: Redirige a `/pago/error`. Sugiere usar otro método de pago antes de que expire el Soft-Lock de 15 minutos.
7. **[Fin]** Fin del flujo de usuario.

### 2. Flujo de Tarea: Reconciliación Asíncrona (Sistema)
**Source:** US-PAY-003

1. **[Inicio]** Servidor de Wompi emite HTTP POST (Webhook) a nuestro servidor con el resultado real.
2. El Backend de Nos Fuimos de Finca recibe la petición.
3. El Backend extrae el hash de los headers y lo valida contra el "Event Secret".
4. Si es válido, actualiza el registro en la tabla `Pago` y actualiza la `Reserva` a Hard-Lock.
5. El Backend retorna HTTP 200 a Wompi.
6. **[Fin]** Dispara evento de notificación (M-10).
