# Entregable 11 (D11): Matriz de Autorización y Seguridad

**Proyecto:** Nos Fuimos de Finca
**Fase:** 4 — Modelado del Sistema
**Estado:** Aprobado

### 1. Alcance y Estrategia
La seguridad se maneja mediante autenticación por JWT asimétrico (RS256). La autorización en el frontend se basa en guardias de rutas según el rol (Claim en el JWT). En el backend, la autorización es por Route Middleware y Row-Level Security (RLS) o filtros de query para garantizar que un Propietario no modifique datos de otra persona.

### 3. Matriz de Seguridad de Endpoints API

| Endpoint API | HTTP | Rol Requerido | Validación a Nivel de Objeto (RLS) |
| --- | --- | --- | --- |
| `/api/v1/fincas` | GET | `Público` | Solo retorna fincas con `estado=aprobada`. |
| `/api/v1/fincas` | POST | `propietario-admin` | ID del Propietario se extrae del JWT, no del Body. |
| `/api/v1/fincas/:id` | PUT | `propietario-admin` | Verifica que `finca.propietario_id == jwt.userId`. |
| `/api/v1/admin/fincas/:id/status`| PATCH | `sys-admin` | N/A (Anulación global del Admin). |
| `/api/v1/reservas/soft-lock` | POST | `turista-guest` | ID del Turista se extrae del JWT. Transacción SQL crítica. |
| `/api/v1/pagos/webhook/wompi` | POST | `Público` | Requiere validación de Firma Criptográfica de Wompi (Event Signature), no requiere JWT. |
| `/api/v1/reservas/:id` | GET | `turista`, `propietario` | Turista ve si `reserva.turista_id == userId`. Propietario ve si `finca.propietario_id == userId`. |

### 4. Vectores de Seguridad Especiales
- **Webhook Spoofing (M-04):** Cualquiera puede hacer un POST a `/webhook`. El backend DEBE calcular el hash de la firma enviada en los encabezados usando la clave secreta (Events Secret) de Wompi, y rechazar la petición (401) si no coincide.
- **Mass Assignment:** Al editar una finca, el Propietario no puede inyectar el campo `estado` en el payload JSON para auto-aprobarse. El backend ignora o rechaza ese campo a nivel del DTO/Serializer.
