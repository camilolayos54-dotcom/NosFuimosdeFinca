### 2. Matriz de Riesgos Unificada (Fase 1 + Fase 2)
Al migrar el proyecto a una arquitectura formal de software (Java/JS/PostgreSQL), la matriz de riesgos evoluciona para cubrir amenazas técnicas reales de ingeniería:

| ID | Tipo | Riesgo / SPOF | Probabilidad | Impacto | Estrategia de Mitigación |
| --- | --- | --- | --- | --- | --- |
| R-01 | Base de Datos (D3) | **Colisión Transaccional (Overbooking Técnico):** Dos JS clients enviando peticiones POST concurrentes a Java pueden reservar la misma finca si no hay control de hilos. | Alta | Crítico | Implementar "Pessimistic Read/Write Locks" en las transacciones de PostgreSQL (Hibernate/JPA) en Java para garantizar control de concurrencia a nivel de BD. |
| R-02 | Infraestructura (D6) | **Outage de PaaS o Base de Datos:** Caída del hosting donde viven el `.jar` de Java o el contenedor de PostgreSQL. | Media | Alto | Contratar un PaaS gestionado robusto (Ej. AWS RDS o Railway Postgres) con backups automatizados diarios, evitando auto-hosting crudo en Fase 1. |
| R-03 | Operativo (D4) | **Fuga de Credenciales (Wompi / WABA):** Un desarrollador hace un push al repo público exponiendo las llaves secretas. | Baja | Crítico | Impedir variables estáticas quemadas en código fuente. Todo el Backend Java debe inyectar secretos a través del sistema de `.env` del host. |
| R-04 | UX / JS (D2) | **Caída de Ventas por LCP lento:** El bundle de JS crece sin control y tarda 10s en cargar en zonas rurales (3G). | Media | Alto | Implementar budget de tamaño de bundle en Vite y rechazar PRs si el JS supera 1.5MB totales. |
