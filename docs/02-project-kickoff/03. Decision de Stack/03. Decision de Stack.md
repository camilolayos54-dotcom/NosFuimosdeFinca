### 2. Stack Tecnológico Oficial (ADR)
| Capa | Tecnología Elegida | Justificación NFR / Decisión de Equipo | ADR de Rechazo (Por qué NO otra cosa) |
| --- | --- | --- | --- |
| Frontend B2C | Vanilla JS / HTML5 / CSS (Vite Builder) | Cumple NFR-01 (Peso ultraligero < 1.5MB). Permite cargar super rápido en redes 3G rurales. | **React / Angular:** Rechazados. Son sobreingeniería para un flujo de captura de reservas de 3 pasos, e inflarían el bundle de JS violando el NFR-01. |
| Backend API | Java (Spring Boot o Javalin) | Cumple NFR-02 (Concurrencia). Excelente gestión del thread pool para frenar bloqueos de inventario (Overbooking). Fuerte control de tipos estáticos. | **Node.js:** Rechazado. Su modelo single-thread es menos nativo para modelar candados de bases de datos y transacciones concurrentes complejas requeridas. |
| Base de Datos | PostgreSQL (Relacional) | Ideal para el control de integridad referencial y locks ACID, necesario para el inventario de fechas compartidas. | **MongoDB / NoSQL:** Rechazado. Un cruce asíncrono de reservas no es eventual; requiere consistencia ACID fuerte en el momento de crear la reserva. |
| Mensajería | WhatsApp Business API | Único canal aceptado por la UX campesina (NFR-03). | **Emails SMS o Web Push:** Ignorados completamente por la demografía rural. |
| Hosting MVP | Railway o Render | Despliegue Dockerizado de Java y Base de datos por un costo menor a $10 USD mensuales. | **AWS EC2 Puro:** Muy complejo de gestionar operativamente en la Fase 1. |

### 3. Implicación de Compuerta de Fase
- **Condición:** Proceed. El equipo cuenta con la directriz clara: Se construirá software real usando un Frontend minimalista y un Backend fuertemente tipado que garantice el manejo del cuello de botella (Overbooking).
