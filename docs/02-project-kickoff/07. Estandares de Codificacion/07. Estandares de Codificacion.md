### 2. Estándares por Capa de Software
| Entorno / Capa | Herramienta | Regla Obligatoria |
| --- | --- | --- |
| Backend Java | Checkstyle | Cumplimiento del estándar de Google Java Style Guide (o similar). Variables en `camelCase`, Clases en `PascalCase`. |
| Frontend JS | ESLint + Prettier | No se aceptarán warnings ni variables no utilizadas. |
| Base de Datos SQL | Nomenclatura Estricta | Tablas y columnas en `snake_case` plural (ej. `reservas`, `id_finca`). Esto previene problemas con dialectos en Java (Hibernate/JPA). |
| API REST | Diseño de Endpoints | Sustantivos plurales, sin verbos. Ej: `POST /api/v1/reservas`. Retorno siempre JSON estandarizado. |

### 3. Automatización y Forzado de Reglas (Enforcing)
| Mecanismo | Acción de Bloqueo |
| --- | --- |
| Pre-commit Hooks (Husky) | Evita que el dev haga commit de código JS mal formateado o código Java que no compile. |
| CI Pipeline (GitHub Actions) | Corre `mvn checkstyle:check` o `eslint`. Si falla, rechaza automáticamente el Pull Request impidiendo el merge a `main`. |
