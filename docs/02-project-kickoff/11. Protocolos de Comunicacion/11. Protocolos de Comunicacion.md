### 2. Matriz de Severidad (Triage) para Producción
Al desplegar una aplicación web custom (Java/PostgreSQL/JS), los incidentes se catalogan así:

| Nivel | Significado | Ejemplo Concreto (Aplicación Fullstack) |
| --- | --- | --- |
| SEV-1 (Crítico) | Caída del sistema transaccional. Pérdida de dinero. | Ej. El servidor Java (Backend) arroja HTTP 500 al guardar reservas o PostgreSQL está inaccesible. |
| SEV-2 (Mayor) | Degradación de servicio. | Ej. El cliente JS carga la página, pero la notificación asíncrona por WhatsApp API a las fincas falla (time-out). |
| SEV-3 (Menor) | Bug de UI o UX no bloqueante. | Ej. El botón de pago Wompi está desalineado o hay un error ortográfico en la landing JS. |

### 3. Rutas de Escalación y SLA
| Severidad | Canal de Reporte / Alerta | SLA de Respuesta Esperado |
| --- | --- | --- |
| SEV-1 | Alerta automatizada (Ej. Sentry/Datadog) + Llamada al DevOps/CTO | < 30 minutos |
| SEV-2 | Grupo de WhatsApp Técnico + Ticket en Backlog con prioridad HIGH | < 4 horas |
| SEV-3 | Tablero de Tickets (Backlog estándar) | Próximo Sprint |

### 4. Protocolos de Desarrollo
- **ADRs (Architecture Decision Records):** Obligatorios si el CTO decide introducir un nuevo framework (Ej. Cambiar Vanilla JS por React, o agregar Redis).
- **Documentación de API:** Obligatorio mantener Swagger / OpenAPI actualizado en Java para no bloquear al desarrollador del Frontend.

### 5. Implicación de Compuerta de Fase
- **Condición:** Proceed. El rigor comunicativo se centra ahora en la estabilidad del servidor y en mantener sincronizado el contrato API entre Frontend y Backend.
