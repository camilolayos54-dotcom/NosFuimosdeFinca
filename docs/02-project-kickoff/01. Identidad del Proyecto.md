### 2. Identidad Nominal y Mandato
- **Codename Técnico:** `nff-platform`

- **Estado de Mandato (Fase 1):** Go
- **Implicación Arquitectónica Principal:** Desarrollo de Software Customizado. Arquitectura desacoplada: API transaccional robusta (Java) y cliente web ligero asíncrono (JavaScript). 

### 3. Hipótesis Central de Valor
La intermediación de reservas rurales asíncronas mediante una máquina de estados centralizada (Backend Java) resuelve el overbooking sin que la finca necesite internet constante, delegando la confirmación de la disponibilidad a un sistema de "Solicitud Pendiente" alertado vía WhatsApp.

### 4. Delimitación de Alcance (Scope Bounds)
**In-Scope (Mandatorio para v0.1.0):**
- Frontend Web en JavaScript para captura de leads B2C y flujo de Checkout. (Justificación: Acceso universal desde cualquier navegador sin descargar apps).
- API Backend en Java para orquestar los estados de reservas concurrentes y manejar la base de datos.
- Notificaciones B2B vía WhatsApp. (Justificación: Las fincas usan WhatsApp asíncronamente; no entrarán a un panel de control web).
- Pago diferido con links asíncronos (Wompi).

**Out-of-Scope (Estrictamente Prohibido para v0.1.0):**
- App Nativa iOS / Android. (Motivo: Costo elevadísimo y no requerido para probar validación).
- Pagos recurrentes integrados en cuenta bancaria. (Motivo: Alta carga de compliance legal; se terceriza el checkout a un link simple de Wompi).
- Dashboard Web Analítico para Fincas B2B. (Motivo: UX innecesaria para el campesino que solo lee WhatsApp).
