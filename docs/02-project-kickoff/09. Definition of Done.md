### 2. Definition of Ready (DoR) — Compuerta de Entrada al Sprint
Un ticket puede comenzar desarrollo si y solo si cumple TODOS los siguientes criterios:
- [ ] Descripción clara de la lógica de negocio (Backend) o el Flujo de Pantalla (Frontend).
- [ ] Criterios de aceptación en formato Gherkin (Given / When / Then).
- [ ] Si involucra BD: Detalle de las tablas afectadas en PostgreSQL.
- [ ] API Contract definido: Request y Response esperados entre el cliente JS y el servidor Java.
- [ ] Estimación de esfuerzo asignada (Story Points o horas).

### 3. Definition of Done (DoD) — Compuerta de Salida al Merge
Un Pull Request puede fusionarse si y solo si cumple TODOS los siguientes criterios:

**Criterios Automáticos (Verificados por CI — GitHub Actions):**
- [ ] CI verde: Compilación de Java exitosa (Maven/Gradle).
- [ ] Tests Unitarios de Java pasan sin fallos.
- [ ] Linter de Frontend (ESLint/Prettier) pasa sin warnings.
- [ ] Sin conflictos de merge contra la rama base.

**Criterios Manuales (Verificados por Revisor Humano):**
- [ ] Aprobación cruzada. El creador del Pull Request NO puede fusionar su propio código.
- [ ] Todos los criterios de aceptación del ticket verificados.
- [ ] Sin `System.out.println()` ni `console.log()` perdidos en el código.

### 4. Implicación de Compuerta de Fase
- **Condición:** Proceed. El proyecto se rige ahora por un estándar de construcción de software puro. Todo nuevo desarrollo debe estar orquestado mediante tickets que respeten el DoR y todo el código pasará por el DoD antes de llegar a los usuarios.
