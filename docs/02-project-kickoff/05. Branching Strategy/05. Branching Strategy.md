### 2. Estrategia de Flujo (Git Flow)
- **Modelo Elegido:** Trunk-Based Development Ligero (Ramas Efímeras).
- **Justificación:** Al ser un MVP de código fuente puro (Java + JS), el commit directo a `main` es peligroso porque puede romper el build (Java Compilation) y tirar Staging. Se requieren ramas efímeras de características (`feat/...`) que pasen por integración continua antes de fusionarse.

### 3. Convención de Commits (Conventional Commits)
| Tipo | Uso | Ejemplo |
| --- | --- | --- |
| `feat:` | Nuevo requerimiento del Backend Java o Frontend JS. | `feat(backend): agrega servicio de lock pesimista` |
| `fix:` | Corrección de bug. | `fix(frontend): repara overflow del modal B2C` |
| `docs:` | Actualización del framework de fases. | `docs(p3): inicia historia de usuario 1` |
| `chore:` | Actualización de dependencias o builds (POM / package.json). | `chore(backend): actualiza spring-boot a 3.x` |

### 4. Reglas de Protección
| Rama | Regla Crítica | Reviews Necesarios |
| --- | --- | --- |
| `main` | Protegida. Solo se avanza mediante Pull Request. Requiere que el Pipeline de Github Actions pase en Verde (Java Build + JS Lints). | 1 (Aprobación visual de otro founder si no hay más devs). |

### 5. Implicación de Compuerta de Fase
- **Condición:** Proceed. Se restableció la burocracia mínima de protección de código, necesaria para lenguajes compilados y evitar caídas en producción.
