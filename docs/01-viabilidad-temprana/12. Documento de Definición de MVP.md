### 2. Hipótesis Central

We believe that **Propietarios de fincas turísticas** will use **Nos Fuimos de Finca** because it solves **la coordinación asíncrona de alquileres y prevención de reservas cruzadas (ineficiencia logística)** better than **WhatsApp + Libretas Físicas**.

### 3. Alcance del MVP

| Capability | Hypothesis Testing Justification |
| --- | --- |
| Módulo de Autenticación Simple (Propietarios, Turistas y Administradores) | Necesario para vincular el inventario a un responsable, los pagos a un comprador, y proveer control centralizado a la plataforma de forma segura (D1). |
| Creación de Ficha de Propiedad (Fotos, Tarifas) | Requisito mínimo para que el turista tenga contexto de lo que está alquilando (P0-D1). |
| Motor de Reservas con Bloqueo Atómico | Funcionalidad central para eliminar las dobles reservas concurrentes (D3). |
| Flujo de Checkout con Integración de Pasarela (Webhook) | Necesario para confirmar la intención real de reserva y formalizar el abono directo al propietario (D2). |
| Manejo de Soft-Locks y liberación de inventario | Mitigación estructural del riesgo técnico de transacciones bancarias expiradas o asíncronas (D5). |
| Panel de Control Administrativo (Básico) | Permite al Administrador gestionar usuarios, ver transacciones globales y brindar soporte técnico. |

### 4. Fuera de Alcance

| Excluded Item | Reason for Exclusion |
| --- | --- |
| Aplicación móvil nativa (iOS/Android) | Retrasa el MVP y aumenta el costo; una PWA (Web App) móvil es suficiente para probar la tracción inicial (D5). |
| Módulo de Calificaciones y Reseñas | No previene el cruce de reservas (nice-to-have post-estadía). |
| Motor de precios dinámicos | Complejidad algorítmica innecesaria; los propietarios rurales prefieren control manual de sus tarifas (D1). |
| Integración iCal con Airbnb/Booking | Requiere desarrollo extra complejo y el segmento objetivo suele rechazar a las OTAs por sus comisiones (D2/D3). |
| Chat embebido en vivo | Complejidad evitable; un enlace directo (handoff) a WhatsApp tras el pago validado resuelve la comunicación logística. |

### 5. Criterios de Éxito

| Métrica | Línea Base | Objetivo | Fecha |
| --- | --- | --- | --- |
| Drop-off de Propietarios en Onboarding | 100% (No existe producto) | < 40% de drop-off | Mes 1 post-lanzamiento |
| Tasa de Reservas Cruzadas en propiedades activas | Desconocida / Alta (P0-D2) | 0% (Absoluto técnico) | Mes 2 post-lanzamiento |
| Conversión de Checkout (Turistas) | 0% | > 15% de usuarios que inician checkout completan el pago | Mes 2 post-lanzamiento |
| Adopción recurrente de dueños | 0% | > 60% de dueños con 1 reserva retienen su calendario activo al mes siguiente | Mes 3 post-lanzamiento |
