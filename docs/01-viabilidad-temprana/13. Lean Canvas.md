# 13. Lean Canvas

### 1. Metadatos del Documento
**Proyecto:** Nos Fuimos de Finca
**Fase:** 1 — Viabilidad Temprana (Early Viability)
**Entregable:** 13 de 18
**Estado:** Aprobado

### 2. Lean Canvas

- **1. Problema:** Altísima fricción en reservas de turismo rural debido a colisiones de inventario (overbooking) causadas por dueños de fincas operando desconectados de internet.
- **2. Segmento de Clientes:** Dueños administradores de fincas (B2B - Baja competencia técnica). Agencias y Turistas informales (B2C).
- **3. Propuesta de Valor Única:** Sincronización asíncrona mediante un backend Java robusto que permite a las agencias vender sin miedo al overbooking, adaptándose al comportamiento offline del dueño de la finca a través de integraciones simples (WhatsApp + Interfaces web JS ligeras).
- **4. Solución (MVP):** MVP de Software Web. Modelo de inversión de oráculo (asumir 100% disponibilidad en el Front JS, gestionar el bloqueo de recursos en el Backend Java y forzar aprobación asíncrona manual).
- **5. Ventaja Injusta (Moat):** Red descentralizada offline operada por un sistema que asimila la falta de conectividad como una feature en su máquina de estados (Backend).
- **6. Canales:** Ventas B2B presenciales a agencias. Intermediación vía links B2C.
- **7. Flujo de Ingresos:** Service Fee de plataforma cobrado al turista.
- **8. Estructura de Costos:** Servidores Cloud (Base de Datos + App Server Java + Frontend Hosting) y Costos API (WhatsApp, SMS, Wompi).
- **9. Métricas Clave:** 
   - Conversión B2C (Checkout).
   - Velocidad de SLA de respuesta B2B (< 2h).

### 3. Alternativas Existentes
- Cuaderno espiral físico y audios de WhatsApp.
- PMS tradicionales pesados que no toleran desconexión rural.

### 4. Implicación de Compuerta de Fase
- **Condición:** Proceed. El Canvas resume un modelo sólido, anclado en software propio (Java/JS) que otorga un foso tecnológico (Moat) real contra intermediarios análogos.
