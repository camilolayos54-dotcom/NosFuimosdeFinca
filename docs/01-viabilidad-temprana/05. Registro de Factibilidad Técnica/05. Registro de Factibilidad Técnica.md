### 3. Reporte de Technical Spike
- **Hipótesis a Probar:** Overbooking catastrófico si múltiples agencias reconectan a internet 4G y lanzan reservas represadas al mismo milisegundo.
- **Diseño de Prueba (Script Desechable):** Script Python de estrés disparando 50 hilos concurrentes asíncronos contra un lock optimista en la BD.
- **Resultados y Hallazgos:** El backend bloquea el 98% (Devuelve HTTP 409 Conflict). El problema es de Experiencia de Usuario: la agencia creerá que vendió el paquete cuando estaba offline, y horas después recibe el 409, causando una crisis B2C con el turista.

### 4. Riesgos Técnicos Residuales
**Riesgo 1:** Colisión de Inventario Asíncrono.
- **Impacto:** Falsa ilusión de venta en las agencias locales que operan en offline.
- **Ruta de Resolución (Pivot/Mitigación):** Cambio radical de arquitectura UX. El sistema JAMÁS aprueba reservas. Todas entran en estado "Pendiente de Veto". El dueño de finca recibe los 50 requests y manualmente oprime "Aprobar" a una.

### 5. Conclusión de Factibilidad
- **Veredicto Final:** Proceed (Con Pivot Operacional).
- **Justificación:** El problema técnico de overbooking se delegó al lado de operaciones y se mitigó por diseño, salvando la viabilidad del proyecto de software.
