# 12. Definición de MVP

### 1. Metadatos del Documento
**Proyecto:** Nos Fuimos de Finca
**Fase:** 1 — Viabilidad Temprana (Early Viability)
**Entregable:** 12 de 18
**Estado:** Aprobado

### 2. Mandato del MVP (Hipótesis a Probar)
| Supuesto a Probar (Desde D11) | Experimento Asignado |
| --- | --- |
| El Turista pagará el Service Fee. | Despliegue de flujo de Checkout real conectado a Wompi. |
| La Finca responde WA en < 2 horas. | Oráculo manual automatizado: Servidor Java notifica vía WA API y espera callback. |

### 3. Factor de Forma y Arquitectura Inicial
- **Form Factor:** Software Web Application (Monolito Liviano).
- **Herramientas a Alto Nivel:** JavaScript (Frontend B2C/B2B ligero), Java (Backend Robusto para lógica concurrente), Base de Datos Relacional.

### 4. Barreras de Alcance (Scope)
**In-Scope (Requerido para el Test):**
- Landing B2C en JS que calcule disponibilidad y cobre mediante link Wompi.
- Lógica Backend en Java que guarde la reserva en estado "Pendiente" y despache la alerta asíncrona por WhatsApp a la finca.
- Interfaz mínima de confirmación para el finquero (vía enlace de retorno o respuesta de WA).

**Out-of-Scope (Prohibido construir ahora):**
- App Nativa iOS / Android (Sobra para probar el MVP).
- Dashboards analíticos complejos B2B.
- Pagos recurrentes automatizados o retenciones de saldo directas (requiere alta ingeniería financiera; Wompi con links asíncronos bastará).

### 5. Criterios de Salida (Exit Criteria)
| Métrica (De D11) | Umbral de Éxito | Acción si Falla (Pivot/Kill) | Acción si Pasa (Proceed) |
| --- | --- | --- | --- |
| Conversión B2C | > 15% click en Pagar | Pivot Comercial (Descontar fee a la Finca). | Aprobar flujo de UX actual e iterar producto. |
| Velocidad B2B | > 80% responden <2h | Kill Project (El campesino no usará el sistema). | Autorizar Fase 4 (Escalamiento y optimización del código). |
